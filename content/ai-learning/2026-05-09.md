## 今日最有感的事

### 把每週維修變成雲端 routine 才是正解

今天最大進展是把「急專大補帖 v2 每週維修」這件事，從『我哪天有空再坐下來檢查』變成 [Claude Code Routines](https://claude.ai/code/routines) 排程的雲端任務，固定每週一上午 9:00（台灣時間）自動跑。

最初版設定其實有兩個明顯問題。第一，prompt 寫太模糊，只說「檢查完整性、提建議」，但什麼叫完整性？哪些欄位？哪些 table？UI 哪幾頁？這種模糊指令丟給 Opus 4.7 1M，每週跑出來的東西會漂得很遠。第二，原本是 Opus 4.7 跑這種「結構化檢查 + 寫報告」任務，這代表 overkill——Sonnet 4.6 速度更快、成本顯著低，這題量級綽綽有餘。

更直白地說，Routine 是 remote agent，跑在 Anthropic 雲端，碰不到我本機檔案系統。所以原本設計「寫成 MD 存我本機」根本做不到——它實際是 push 到 GitHub branch、再開 PR。把這個 mental model 校正過來之後，整個工作流就清楚了：

1. **雲端 routine（每週一 9:00）**：clone repo → 跑檢查 → 寫 `reports/weekly-YYYY-WNN.md` → push branch → 開 PR
2. **本機 Claude Code（我坐下來時）**：看 PR / 報告 → 拿著清單實際動手修

真正值得看的是這個分工裡有個優化點：**雲端能直接修的就讓它直接修，不要全推回本機**。比如 figures 路徑格式錯、tsc/build error、過期依賴 bump，這些雲端 SQL 或 npm 一句話就能搞定，根本不該排隊等我親手敲。需要醫學判斷的（題目敘述、答案對錯、rationale 內容）才送回本機 triage。

清完 prompt、換成 Sonnet 4.6、把 MCP 砍到只剩 [Supabase](https://mcp.supabase.com/mcp) 跟 [Vercel](https://mcp.vercel.com)（Gmail、Calendar、Drive、PubMed 這次任務都用不到）後，手動觸發第一輪 W19 維修，51 分鐘跑完整輪：4 套件升級上線、刪測試 feedback、更重要的是——抓到了真 BUG。

#### 真正的 reframe：「多選題」其實是「申複放寬計分」

Routine 報告原本標 11 道為「多選題 UI 判分問題」，我看了之後把它 reframe：這些題目不是要使用者勾多個答案，而是**考試考完後檢討發現可能不只一個正解，所以申複放寬計分**。意思是 UI 流程照舊單選，但判分邏輯只要使用者選的答案 ∈ `answer_key.split(',')` 就算對。

這個 framing 一改，要查的東西完全不同——不是看 UI 顯示，而是 grep 判分代碼有沒有 `split + includes`。grep 下去就抓到 `pages/ExamResult.jsx` 同檔案兩條 path 共存，line 74 還在用 `userAnswer === q.answer_key` 直接比對，會把 11 題使用者選對的結果記成錯。最後 4 處全修上線。

這件事讓我學到：**routine 報告本身的 framing 不一定對，但只要你願意 reframe，它幫你把搜尋範圍縮到 5 題裡，就是巨大的省時**。它不需要替我下醫學判斷，它只要把『可疑的東西』收斂到夠小，剩下的我來。

### DHH 半年內從「不用 AI」到 agent-first

下午看了 The Pragmatic Engineer 的這集：[DHH's new way of writing code](https://www.youtube.com/watch?v=JiWgKRgdgpI)（1h47m）。半年前他在 Lex Fridman 上才說『我不用 AI 寫 code、所有 code 都自己打』，這集 180 度大轉彎——現在 agent-first，幾乎不再手寫 code。重點是他**品味跟 craft 標準沒退讓**，只是把實作這層 outsource 掉。

更直白地說，DHH 提到 AI 對資深 vs junior 工程師的衝擊是不對稱的：資深被放大、junior 反而變難。這個觀察跟我自己工作流的演進完全對得上——能下對的 prompt、能 reframe routine 報告（像今天那 11 題）、能在 BUG fix 之後問「這個 PR 真的解決根因嗎」，靠的全是過去累積的判斷力。AI 沒讓判斷力變便宜，它讓判斷力變得更值錢。

順帶記一下技術細節：1h47m 英文 podcast 用本地 transcribe skill 跑，Stage 1 yt-dlp 抓音訊 + 元資料、Stage 2 whisper turbo 約 3 分鐘出英文逐字稿、Stage 3 切 6 段派 12 個 Haiku sub-agent 並行（6 校正 + 6 翻譯）。這條 pipeline 其實就是 DHH 講的那種 agent-first 架構的縮影——我下決策、agent 並行幹活。

## 今日收集的資源

**Claude Code Routines（每週排程入口）**
- 連結: https://claude.ai/code/routines
- 一句說明: 排雲端 cron，這次拿來綁急專大補帖 v2 的每週一 9:00 維修檢查。

**單一 Routine 設定頁**
- 連結: https://claude.ai/code/routines/{ROUTINE_ID}
- 一句說明: 改 model、prompt、MCP allowlist 都在這頁，今天把 Opus 4.7 換成 Sonnet 4.6 就是從這裡。

**Claude.ai Connectors 設定**
- 連結: https://claude.ai/customize/connectors
- 一句說明: routine 能用哪些 MCP 受這裡的 connector 啟用狀態決定，要拔掉用不到的 server 從這邊管。

**Supabase MCP**
- 連結: https://mcp.supabase.com/mcp
- 一句說明: routine 要 `list_tables` 探 schema、`execute_sql` 抓題目完整性都靠這條，是急專大補帖維修的主力。

**Vercel MCP**
- 連結: https://mcp.vercel.com
- 一句說明: 跑完維修後直接看 build 狀態、抓 deploy log，配合 Supabase MCP 形成「DB → 部署」全鏈路。

**Drive MCP（這次砍掉但留檔）**
- 連結: https://drivemcp.googleapis.com/mcp/v1
- 一句說明: 急專維修用不到，從 routine allowlist 拔掉；但本機 Claude Code 做逐字稿上傳 Google Docs 時還會用。

**Routine GitHub repo（範例占位）**

**Routine GitHub repo（小寫 placeholder）**

**The Pragmatic Engineer：DHH's new way of writing code**
