## 今日最有感的事

### 一、AI 把臨床判斷寫得像有 source，要靠自己 audit 才看得出來

今天在跑嘉明湖醫療專案，問題是兩位 CHD（先天性心臟病）小孩能不能上 2850m 向陽山屋。Agent 列了六條放行 cutoff：「修補後 ≥ 2 年」「PASP ≤ 30 mmHg」「海平面 SpO₂ ≥ 95%」「NYHA I-II」之類，條列分明、像是直接抄自某本指引。我隨手問一句「這資料哪裡來的？」agent 才老實承認：那六條全是它自己合成的，不是任何指引明文。

這代表 LLM 的天然輸出風格就是把臨床判斷寫得「像條列指引」，但其實後面沒有 source。對我這種會直接拿去簽家長知情同意書的使用者來說特別危險——讀起來太順、太像 UpToDate 摘要，會讓人下意識相信。

接下來換打法：用 OpenEvidence 的 `oe_ask` 重跑兩條 query（兒童 CHD 中海拔登山評估、兒童 Diamox 劑量），再用 `audit-oe` skill 派 5 個 haiku agent 並行去 PubMed 抓全文 cross-check。三件事被打臉：

1. **沒有兒童 PASP 上山 cutoff 的文獻共識**——之前訂「30 mmHg」是憑印象寫的。真正有共識的是成人 60 mmHg，兒童版根本沒有指引切點。
2. **「修補後 2 年」不存在於任何指引**——agent 從「術後 6 個月禁高山」這條保守往上加碼，加到一個聽起來合理的數字而已。
3. **Diamox 劑量抓到 cross-source contradiction**：CDC Yellow Book 寫 1.25 mg/kg q12h（max 125 mg），WMS 2024 寫 2.5 mg/kg q12h，**剛好差兩倍**。OE 的答案把這條歸給 WMS，但結構化 citation spans 並沒有單獨 attach 這個數字到任一篇 paper，audit 才抓到這個漏網之魚。

真正值得看的是這個流程的三段：先讓 OE 跑帶 citation 的答案、再用 audit-oe 把每條引用拉去 PubMed 重驗、最後找跨來源矛盾。一條臨床問題如果只走第一段，就會被「OE 也有 citation」這個外觀騙過。

更直白地說：AI 給的引用要再分一層——「OE 有引這篇」不等於「這篇真的支持這句話」，更不等於「不同來源彼此一致」。寫嘉明湖計畫書時，這條直接影響我要不要在家長知情同意書上把 Diamox 兒童劑量寫成單一數字，還是兩家來源都列出來說明分歧。最後選了後者。

### 二、bug 不要繞過，去修根因

早上抓 Rex Chen 寫的 Claude Code 輔助工具 FB 貼文，agent 第一輪用 `fetch-social.js` 抓到的卻是「FB 動態消息側欄」，不是貼文本體。它當下偷懶換工具——直接 call Playwright MCP 跑一次 Chromium，剛好這篇是公開貼文、沒登入也看得到，矇對了。

我立刻抓出兩件事：(1)「你不是用 Edge 嗎，怎麼又開 Chrome？」(2)「你不是要修 fetch-social 嗎，怎麼換工具繞過？」

這代表 agent 做了兩個錯誤決策。第一，它把「fetch-social = 用 Edge」的設計搞混——這 script 真實設計是「啟動裸 Chromium，把 Edge 的 cookies 從 sqlite 撈出來灌進去」，因為 Edge 本體在跑時 user data dir 鎖死，Playwright 根本開不了 Edge profile。第二，它用「換工具繞過」代替「修根因」，下次同情境一定重現。

修完抓到真 bug：FB share URL（`/share/p/...`）會 redirect 到 `/story.php`，貼文本體渲染在 `[role="dialog"]` 裡而不是 `[role="main"]`，原本 script 抓 `main` 只會拿到動態消息側欄。改成等 dialog 出現再讀 innerText、加 `--headed` debug 旗標讓視窗可見，才驗到 cookies 確實有掛上、登入態真的有進去。

順手也清了 memory——之前 `feedback_social_fetch.md` 寫「2026-05-07 已驗證抓得到」是誤報，那次抓到的「ER Bear 朋友列表」其實也是動態側欄，當時把「看到登入態」誤認為「URL 內容對」。這種 memory 自我污染比 bug 本身還危險，因為之後 agent 會直接拿來當「已驗證」依據，bug 就被埋進 trust chain。寫 memory 的時候要驗到「輸出內容真的對」才能寫 verified，不是「沒報錯」就算數。

### 三、急診薪資自動 pipeline 又多吃進一家醫院

下午 13:15 cron 觸發 er-salary 自動 fetcher，跑完 Supabase 多了 **中醫大新竹** 一筆（15 班 / 45 萬，PPH 1.8-2.1，10 位急專滿班，今年調升 10% 以上）。順手做兩件事：(1) 用 supabase MCP 走管理權限把四個 alias 寫進去（中國醫新竹／中國醫新竹分院／中醫新竹／中國醫藥大學新竹附設醫院），讓未來 FB 上有人寫別名也能被 AI 解析 match 到；(2) 修 mobile RWD——dashboard 在手機上要往右滑超遠才看得到薪資，根因是 desktop CSS 設了 `table { min-width: 1200px }`、mobile @600px 沒 override，加一條 `min-width: unset` + 改成卡片式 layout 就解掉。

順手抓到一個 RLS 警訊：anon key **能 INSERT 不能 UPDATE**，PATCH 回 200 但 `content-range: */0` = 0 rows updated（RLS 靜默擋）。這個之後要修——目前正式更新全靠 supabase MCP 的管理權限，不適合給 fetcher 直接寫，應該補一條給自動化用的 service role 路徑、或者把 update policy 重新設計過。
